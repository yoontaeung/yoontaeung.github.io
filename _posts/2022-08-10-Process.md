# L3. Processes

L3. Processes

Process: an instance of a running program

A process includes:

Images(code, data, stack, heap)

Process context(program context(data registers, PC, SP), kernel context(pid, gid..))

프로세스 이미지에는 code, data, heap, stack 이 있다. stack 은 위에서 아래로 늘었다 줄어들었다 하며 데이터를 저장하는데, 지역변수나 함수의 return 주소 등을 저장한다. heap은 dynamically allocated memory(malloc) 가 있으며, data에는 initialized or uninitialized data가 있다.

**Process switching**

Multi processing OS. 여러 개의 프로세스가 동시에. CPU는 타임 퀀텀을 각 프로세스에 주면서 run, block, ready 등의 state 변화를 거친다. 결국 특정 순간 하나의 process만 수행되고 scheduler는 현재 process를 멈추고 다른 걸 골라서 수행. 이런 상황에서 process switching이 발생. Runtime에 두 개의 process가 CPU에 선택.
어려운 이유는 기존 프로세스가 끝난게 아니고 잠시 멈춘 것. 멈췄을 적 모든 state를 복원해야 한다. 언제 switch? Mode의 switching을 먼저 해야함(process switching) 과 다름. OS가 어떤 역할?

스케줄링을 하기 위한 기술적으로 필요한 운영체제의 requirement. Multi-processing 유닛. 런타임에 두 개의 프로세스가 cpu가 선택. 기존 process 가 블록 될 때 다른 프로세스가 수행함. 기존 프로세스가 끝난게 아니라 멈춘 것. 멈췄을 적 모든 state을 복원해야함. 언제 스위치할 것인가? 모드의 스위칭(유저, 커널 모드)과 프로세스의 스위칭의 차이? 어떤 일을 해야하는지?

**운영체제가 동작하는 원리. ******
CPU가 유저모드에 돌 수 있고 중요한 커널 코드를 수행하는 모드로 돌 수 있음. 세 가지 경우(큰 틀) 모드 스위칭 발생. 1) interrupt가 걸릴 경우. Interrupt가 오면 모드를 바꾼다. 해당 interrupt handler를 돌리기 위해. Timer interrupt가 발생할 때 역시. 2) internal trap. 예상치 못한 상황. 커널이 page fault exception 핸들링을 함. 3) system call. 우리가 짠 코드의 시스템 콜 함수를 부를 적. 의도적으로 커널 모드 부름.
무조건 모드 점프를 할 수 없다. 임의의 위치이기 때문에 현재 state를 잘 기록해야함. (register, flag 등을 어디에 저장해야함). **Processor state을 반드시 저장.**
위 세가지 경우 발생시 모드 switching. 모드 스위칭을 기본으로 process switching. 모드 스위칭보다는 무거운 복잡한 작업. Mode switching이 선행되어야 함. 모드 스위칭은 빈번. 그렇다고 반드시 프로세스간 스위칭이 발생하는건 아님.

“Process switching” may occur as the result of mode switching!

5 mode operation(ready, running, block 그림) 생각하면 됨

save the ‘program context’(registers) → update the state of the current process → move the PCB to the appropriate queue → select another process to execute → move its PCB from the appropriate queue → update the state of the selected process to running → update necessary memory management structures → restore the program context of the newly selected process

The process switch, which involve a state change, requires considerably more effort than a mode switch.

Terminal 에 cat fileA라는 명령어 입력. fileA의 내용을 보겠다. Shell은 그 다음에 $를 내보냄. 이 과정이 수행되는 flow를 보면 우리가 터미널에 앉아있고 cat fileA를 입력함. Shell은 많은 종류가 있음. Shell은 우리와 대화하는 프로세스다. Interactive 한 프로세스. Shell이 명령어를 받으면 shell이라는 프로세스가 우리의 명령어로 만들어진 cat 을 수행하는 프로세스를 만든다. Shell이 cat 프로세스를 만든다. Shell 프로세스가 자기와 똑 같은 child를 만든 후 그것을 cat으로 바꾼다. Shell은 cat을 만들었기 때문에 next line에 $를 터미널에 보내고 임무 수행 종료. Cat은 fileA를 디스크에서 읽어서 화면에 뿌려준다. 어떻게 명령어가 수행되는지 내부 프로세스 설명. 철저하게 cat은 shell의 child가 된다. 어떤 명령을 shell에 치더라도 그 명령어로 구성된 process가 만들어진다. Shell 프로세스가 우리와 어떻게 대화하는지에 대한 틀을 보여주는 그림. 시스템을 부팅하면 처음에 만들어지는게 root프로세스다. 최종적으로 init이라는 pid1번. init프로세스는 모든 프로세스의 원조격. 적당한 시점에 shell을 folk. 이 방식이 전통적인 unix라는 os에서 프로세스의 생성 틀.

Exec를 써서 어떤 의미있는 작업을 할 수 있는가? 비로소 완벽한 새로운 프로세스 구현 可. Fork와 exec를 같이 쓰면 어떨까.
새로운 child에게 pid=105를 줬다. Fork가 끝나고 exec 수행. 그 순간부터 105 프로세스안에 모든 이미지가 bin/ls 코드로 바뀐다. 결론적으로 두 개의 프로세스가 존재. Pid, 프로세스 이미지가 전혀 다른 두 개의 프로세스 생성 완료.
42/
Fork한 child 프로세스가 잘 수행할 때까지 parent 는 기다린다. shell이라는 큰 프로세스를 간단하게. 적절하게 fork, exec계열 시스템 콜을 잘 활용.
43/ exit( )
Exit은 주변 정리한다. 깨끗하게 process 정리. Child가 끝나면 exit 상태를 parent한테 전달한다. Exit 인자로 전달. _exit()는 clean up안하고 곧바로 커널로.
44/
Exit( ) 하면 다 정리하고 kernel로. 아니면 직접 _exit( ) 해서 오른쪽 과정안하고 kernel로.
45/
Parent가 fork한다. Child가 얼마 수행 후 exit 수행. Exit(1) 1이 exit status. parent에서 wait 콜을 한다. Wait 콜을 통해서 child가 주는 state을 s로 받는다. Child가 다 할 때까지 parent가 기다린다. 결국 wait 콜을 써서 explicit한 동기화가 가능하다.
46/
Cooperation 한다? 첫 번째로 통신을 할 수 있어야 한다. 데이터 교환. 그리고 어떤 순서를 가지고 동기화를 하느냐. 동기화 이슈. 결국 process가 coordination한다는건 통신과 동기화가 잘 지원. 프로그램 혹은 컴파일러가 제공. Pl이 통신과 동기화에 대한 기능을 제공해 컴파일러가 알아서 잘 바꿔줌(application programming). 커널이 제공하는건 IPC. 즉 프로세스간 기능만 제공.
47/
Producer, consumer problem. 병렬 프로그램의 예시. 공유 데이터를 활용해 두 개의 프로그램이 공유. Producer는 데이터를 계속 만들어 공유 데이터에 저장. Consumer는 공유 데이터를 가져와 쓴다.
n개의 element를 갖는 버퍼가 있다. In, out은 인덱스. 이걸 바탕으로 producer는 버퍼가 꽉차면 기다림. Condition check 해서 버퍼가 빌 떄까지 no-op. consumer는 소모한다. 비었으면 기다림. Buffer에 있는 인덱스를 가지고 동기화.

48/ interprocess communication
통신이 자연스럽게 동기화까지 구현. IPC라고 한다. 여러 개의 IPC 모델이 있다. 단일 호스트 안에서 IPC도 가능. 하나의 컴퓨터안에 두 개의 process. 또한 컴퓨터1, 2끼리 프로세스간 통신도 있다.
일반적으로 client server model이 있다. Web server, web client. Client는 서비스 요청을 함. 이게 IPC의 구체적인 모습
49/ types of unix IPC
unix운영체제에서 제공하는 IPC. 전통적인 basic IPC(pipes, fifo),

50/ pipe
Pipe는 단일 컴퓨터에서만. 우리가 어떤 파이프를 설정하는데, 파이프는 커널에 만들어지고 (관처럼 생각) 유저 프로세스에서 파이프에 데이터를 쓰거나 파이프의 데이터를 읽는다. 실제로는 parent가 child를 만들고 동기화를 할 경우에 파이프를 쓴다. Parent가 파이프를 설정하고 fork후 child 생성. Parent, child가 쓸 수 있는 파이프가 있고 parent가 child에게 데이터를 주려면 pipe 에 write을 하고 child는 read. 반대도 마찬가지. Parent, child간의 통신.
51/ FIFO
가장 큰 문제는 parent와 child에서만 쓸 수 있다. 또 pipe가 permanent하지 않아 basic pipe의 한계. 이런 문제를 향상하기 위해 FIFO(named pipe) 등장. 커널에 FIFO 파이프를 항상 유지. 프로세스가 죽어도 존재하는 고정된 pipe. 이걸 통해 process 두개가 데이터 주고받음. 임의의 두개의 프로세스가 named pipe를 통해 데이터 주고받는다. 임의의 두개의 프로세스(부모 자식간이 아님)간 데이터 교환. basic보다 generic하다. 하지만 이는 단일 호스트 프로세스간 사용할 수 밖에 없다.
52/ system v IPC
더 고급기능을 가진 IPC. ATnT에서 만든 system V IPC 기법. IPC facilities. Message passing, shared memory, semaphores를 통해 더 유연하게 데이터 교환, 동기화. 구체적으로 알 필욘 없다. 단일 호스트 내 두 개의 호스트가 message를 주고 받는다. 두 개의 프로세스가 메모리를 공유한다. 같은 호스트에 있는 두 개의 호스트가 이미지 공유. 공유의 이슈가 있다보니 동기화 문제를 해결해야함. 때문에 semaphore 기능. But 이런 기능을 호스트 안에서만 썼다.
53/ client server IPC
컴퓨터망에 연결된 두 개의 컴퓨터가 서로를 인지해 데이터 교환하면 좋겠다. Client server IPC등장. 한 쪽 컴퓨터는 서비스 역할. 다른건 서비스 요청. Client server가 pair up. For example, mail server, web server. 통신이 가능해야 한다. 다른 프로세스가 통신해서 데이터 교환.
54/ bsd socket interface
소켓. 두 개 모두 socket API를 가짐. 통신 가능. 소켓 api에서 제공하는 적절한 함수를 사용. OS, 하드웨어가 달라도 socket api를 쓰면 데이터 교환 可.
55/ what is socket?
인터넷이란? 핵심은 internet layer. 인터넷 레이어를 구성하는 기술 중 하나가 Internet Protocol. 하드웨어 컴퓨터는 하나의 고정 ip를 할당받고 이거 위에 여러 개의 transport layer를 구성한다. 어떤 경우는 TCP or UDP위에 app돌아감. 우리가 쓰는 통신은 physical link layers, 위에 IP, transport layer, app. TCP/IP가 거의 모든 컴퓨터에 구현. 내 프로세스가 바다를 건너서 통신하고 싶다면 서로 인지해야함. 해당 컴퓨터의 IP를 알고 그 프로세스를 port number로 정한다. 해당 컴퓨터의 IP와 port number를 알면 특정 可. 그 다음 transport를 TCP? Or UDP?
56/ connectionless udp socket
상대방 컴퓨터 인지 후 어떻게 데이터를 주고 받는가? Client가 서비스 요청. 서버가 돌다가 어느 시점에 특정 클라이언트가 연결 요청(UDP로). 클라이언트는 이미 알고 있는 port, IP로 connection 요청. 그 후 데이터 교환. 두 번째 클라이언트가 요청하면 연결해주고. 서버는 클라이언트로부터 커넥션 요청이 오면 데이터 교환. Udp의 특징은 connectionless. 프로토콜 자체가 reliable하지 않다. 데이터 그냥 보냄. 운 좋으면 받는거. 그래서 unreliable. 대신에 빠르다. 가면 그만 안가면 그만. 우편 배달. 편지에 모든 내용이 다 담겨있음. 수신자 주소, 발신자 주소, 데이터. 편지가 중간에서 사라지면? 손실의 가능성이 있다. 또 order가 없다. 순서가 항상 guaranteed 되는게 아님.
57/ TCP
앞 슬라이드와 다른 몇 가지. 우선 서버에 parent, child가 있다. 클라이언트가 실선으로 연결. 임의의 클라이언트가 child서버랑 1대1로 연결. 연세대 서버 들어가면 parent 서버를 fork해서 child 만듦. 1번쨰 클라이언트만 전담한다. 두 번째 클라이언트가 요청하면 또 fork. 천 명이 요청하면 천 개의 child프로세스 만든다. UDP와 다른 점이 전담 서버 프로세스이기 때문에 실선. 완벽히 reliable. 마치 전화할 때. 순서도 지켜짐. Tcp based의 소켓. 다음 렉쳐에서 문제를 해결.
